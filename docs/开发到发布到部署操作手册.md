# AI Monitor 从开发到发布到部署操作手册

这份文档面向第一次做“修改代码 -> 本机测试 -> 打包 -> 部署到其他机器”的使用者。

目标是让你明确三件事：

1. 平时应该改哪里
2. 什么时候测试、怎么测试
3. 怎么把开发机上的成果做成一个可部署版本，再装到另一台机器上

## 1. 先理解这套目录分别是干什么的

你现在的工程不是只有一个目录，而是几部分一起构成系统：

- `ai-monitor-backend/`
  - Go 后端
- `ai-monitor-service/`
  - Python 算法调度服务
- `ai-monitor-frontend/`
  - Vue 前端源码
- `infer-server/infer-server/`
  - C++ 推理服务源码
- `ZLMediaKit/`
  - 视频流服务
- `ai-monitor-release/`
  - 发布包工作区，不是业务源码目录

最重要的一句话：

- 日常开发时，改的是 `ai-monitor-backend/`、`ai-monitor-service/`、`ai-monitor-frontend/`、`infer-server/infer-server/`
- 发布打包时，用的是 `ai-monitor-release/`
- 目标机器运行时，使用的是打包后的产物，不应该再直接改开发源码目录

## 2. 这套发布方案的基本思路

这套方案把内容分成三类：

### 2.1 开发源码

在开发机上修改：

- Go 后端源码
- Python 服务源码
- 前端源码
- C++ 推理服务源码

### 2.2 发布包

通过 `ai-monitor-release/scripts/package-release.sh` 生成。

打包后会形成一个版本目录，里面包含：

- `bin/`
- `python/`
- `frontend/`
- `models/`
- `audio/`
- `config/`
- `sql/`
- `systemd/`
- `scripts/`

### 2.3 目标机器上的现场数据与配置

这部分不跟着版本覆盖：

- `/etc/ai-monitor/`
  - 配置文件
- `/var/lib/ai-monitor/`
  - 数据库、快照、运行期数据
- `/opt/ai-monitor/releases/<version>/`
  - 每个版本的程序目录
- `/opt/ai-monitor/current`
  - 当前生效版本的软链接

## 3. 日常开发时应该怎么做

### 3.1 改代码

你平时应该直接改这些源码目录：

- Go：`/home/hzhy/ai-monitor-backend`
- Python：`/home/hzhy/ai-monitor-service`
- 前端：`/home/hzhy/ai-monitor-frontend`
- Infer：`/home/hzhy/infer-server/infer-server`

不要把业务改动直接写进 `ai-monitor-release/bin` 或 `ai-monitor-release/python/app` 这类“打包产物目录”，因为这些目录会在重新打包时被覆盖。

### 3.2 本机开发测试

建议每次改完先在开发机本机测通，再考虑发布。

#### Go 后端测试

```bash
cd /home/hzhy/ai-monitor-backend
PATH=$PATH:/home/hzhy/go/bin GOPROXY=https://goproxy.cn,direct go build ./...
```

如果要本机直接运行开发版本：

```bash
cd /home/hzhy/ai-monitor-backend
./start.sh
```

#### Python 服务测试

```bash
cd /home/hzhy/ai-monitor-service
python3 -m py_compile config.py voice_alarm.py alarm_manager.py infer_client.py main.py
```

如果要直接运行：

```bash
cd /home/hzhy/ai-monitor-service
python3 main.py
```

#### 前端测试

开发态：

```bash
cd /home/hzhy/ai-monitor-frontend
npm run dev
```

生产构建测试：

```bash
cd /home/hzhy/ai-monitor-frontend
npm run build
```

#### C++ Infer Server 测试

如果你改了 Infer 源码，至少确认能编译：

```bash
cd /home/hzhy/infer-server/infer-server
cmake -S . -B build
cmake --build build --target infer_server -j$(nproc)
```

### 3.3 联调测试建议顺序

在开发机上联调时，建议按这个顺序：

1. 启动 ZLMediaKit
2. 启动 Infer Server
3. 启动 Python 服务
4. 启动 Go 后端
5. 启动前端开发服务
6. 在页面上测试摄像头、任务、告警、截图、语音、上传

## 4. 什么时候可以准备发版

建议满足以下条件再打包：

- 本机功能已经测通
- 前端 `npm run build` 成功
- Go `go build ./...` 成功
- Python 基本语法检查通过
- 如果改了 Infer，C++ 编译通过
- 你知道这次发布对应的版本号，例如 `2026.04.10-rk3576-v2`

## 5. 开始打包前，你要先准备什么

### 5.1 检查 `ai-monitor-release` 里的模板

重点看：

- `config/backend.env.example`
- `config/python.env.example`
- `config/infer.env.example`
- `config/zlm.env.example`
- `config/server.json.example`
- `config/zlm.config.ini.example`

如果这次版本引入了新的配置项，先把这些模板补齐。

### 5.2 修改版本号

你至少要更新：

- `ai-monitor-release/VERSION`
- `ai-monitor-release/manifest/VERSION`

建议两处保持一致。

## 6. 在开发机上打包

### 6.1 进入发布工作区

```bash
cd /home/hzhy/ai-monitor-release
```

### 6.2 执行打包命令

例如：

```bash
./scripts/package-release.sh 2026.04.10-rk3576-v2
```

如果你不传版本号，脚本会默认读取：

- `manifest/VERSION`

### 6.3 打包脚本做了什么

它会自动做这些事情：

1. 编译 Go 后端
2. 构建前端 `dist`
3. 收集 Python 服务代码和 `venv` 或 wheels
4. 编译 `infer_server`
5. 复制 ZLMediaKit 二进制
6. 复制模型目录
7. 复制音频目录
8. 组装成最终版本目录
9. 生成校验文件 `manifest/checksums.sha256`

最终产物默认在：

- `ai-monitor-release/dist/ai-monitor-release-<version>/`

例如：

- `ai-monitor-release/dist/ai-monitor-release-2026.04.10-rk3576-v2/`

### 6.4 打包完成后你应该检查什么

进入产物目录确认这些目录已经存在：

- `bin/`
- `python/`
- `frontend/`
- `models/`
- `audio/`
- `config/`
- `sql/`
- `systemd/`
- `scripts/`
- `manifest/`

## 7. 打包后如何交付到另一台机器

你有两种常用方式：

### 7.1 直接复制目录

```bash
scp -r /home/hzhy/ai-monitor-release/dist/ai-monitor-release-2026.04.10-rk3576-v2 hzhy@目标机:/home/hzhy/
```

### 7.2 先压缩再传

```bash
cd /home/hzhy/ai-monitor-release/dist
tar -czf ai-monitor-release-2026.04.10-rk3576-v2.tar.gz ai-monitor-release-2026.04.10-rk3576-v2
scp ai-monitor-release-2026.04.10-rk3576-v2.tar.gz hzhy@目标机:/home/hzhy/
```

到目标机后再解压：

```bash
cd /home/hzhy
tar -xzf ai-monitor-release-2026.04.10-rk3576-v2.tar.gz
```

## 8. 目标机器首次部署怎么做

以下步骤是“新机器第一次安装”的流程。

### 8.1 先确认基础运行时已经具备

这部分不属于你的业务发布包，需要目标机预先准备好：

- RKNN Runtime / MPP / RGA
- `/opt/ffmpeg-rk`
- `jemalloc`
- `sqlite3`
- `nginx` 或 `caddy`

### 8.2 进入发布包目录

例如：

```bash
cd /home/hzhy/ai-monitor-release-2026.04.10-rk3576-v2
```

### 8.3 执行安装脚本

```bash
./scripts/install.sh
```

这个脚本会做这些事情：

1. 把版本内容复制到 `/opt/ai-monitor/releases/<version>/`
2. 创建现场配置目录 `/etc/ai-monitor/`
3. 创建现场数据目录 `/var/lib/ai-monitor/`
4. 创建 `/opt/ai-monitor/current` 软链接
5. 如果目标机还没有配置文件，就从 `config/*.example` 复制一份

### 8.4 初始化数据库

首次部署时执行：

```bash
./scripts/init-db.sh
```

它会使用：

- `sql/schema.sql`
- `sql/seed_base.sql`

默认数据库位置是：

- `/var/lib/ai-monitor/aimonitor.db`

## 9. 首次部署后必须修改哪些配置

安装完成后，重点修改：

- `/etc/ai-monitor/backend.env`
- `/etc/ai-monitor/python.env`
- `/etc/ai-monitor/infer.env`
- `/etc/ai-monitor/zlm.env`
- `/etc/ai-monitor/infer/server.json`
- `/etc/ai-monitor/zlm/config.ini`

### 9.1 你最常需要改的配置

#### `backend.env`

主要关注：

- 数据库路径
- Python 服务地址
- ZLM 地址
- 模型目录
- 快照目录
- 音频目录

#### `python.env`

主要关注：

- 数据库路径
- Infer 服务地址
- ZMQ 地址
- 快照目录
- 音频目录

#### `infer/server.json`

主要关注：

- `http_port`
- `zmq_endpoint`
- `streams_save_path`
- 各模型路径

#### `zlm/config.ini`

主要关注：

- API secret
- `ffmpeg.bin=/opt/ffmpeg-rk/bin/ffmpeg`

### 9.2 一个关键原则

现场部署时，优先改 `/etc/ai-monitor/` 里的配置，不要去改 `/opt/ai-monitor/current/` 里的程序源码或模板。

## 10. 如何安装 systemd 服务

进入发布包目录后，把服务文件复制到系统目录：

```bash
sudo cp systemd/zlmediakit.service /etc/systemd/system/
sudo cp systemd/infer-server.service /etc/systemd/system/
sudo cp systemd/ai-monitor-python.service /etc/systemd/system/
sudo cp systemd/ai-monitor-backend.service /etc/systemd/system/
```

然后执行：

```bash
sudo systemctl daemon-reload
sudo systemctl enable zlmediakit.service
sudo systemctl enable infer-server.service
sudo systemctl enable ai-monitor-python.service
sudo systemctl enable ai-monitor-backend.service
```

启动顺序建议：

```bash
sudo systemctl start zlmediakit.service
sudo systemctl start infer-server.service
sudo systemctl start ai-monitor-python.service
sudo systemctl start ai-monitor-backend.service
```

## 11. 如何配置 Nginx

站点模板在：

- `nginx/ai-monitor.conf.example`

它的作用是：

- `/` -> 前端 `dist`
- `/api/` -> Go 后端
- `/snapshots/` -> Go 后端
- `/live/` -> ZLMediaKit

常见做法：

```bash
sudo cp nginx/ai-monitor.conf.example /etc/nginx/conf.d/ai-monitor.conf
sudo nginx -t
sudo systemctl restart nginx
```

## 12. 部署完成后怎么验收

### 12.1 先跑健康检查

```bash
./scripts/health-check.sh
```

它会检查：

- Go 后端
- Python 服务
- Infer 服务
- ZLM
- 对外入口

### 12.2 再做人工功能验证

建议至少检查：

1. 页面能否打开
2. 摄像头能否添加
3. 推流能否启动
4. 实时画面能否预览
5. 任务能否启动
6. Python 是否收到推理结果
7. 告警是否落库
8. 快照是否落盘
9. 语音报警是否正常
10. 报警上传是否正常

## 13. 以后发新版本时怎么升级

### 13.1 在开发机上完成修改和测试

重复前面的流程：

1. 改源码
2. 本机测试
3. 打包新版本

### 13.2 把新版本传到目标机

例如传到：

- `/home/hzhy/ai-monitor-release-2026.04.20-rk3576-v3/`

### 13.3 执行升级脚本

进入新版本目录后执行：

```bash
./scripts/upgrade.sh
```

这个脚本会做这些事情：

1. 备份数据库
2. 停服务
3. 安装新版本
4. 执行数据库初始化/迁移逻辑
5. 启动服务
6. 跑健康检查

## 14. 如果升级失败，怎么回滚

先看历史版本目录：

```bash
ls /opt/ai-monitor/releases/
```

然后执行：

```bash
./scripts/rollback.sh 旧版本号
```

例如：

```bash
./scripts/rollback.sh 2026.04.10-rk3576-v2
```

如果需要，还可以把数据库从备份目录恢复。

数据库备份通常在：

- `/var/lib/ai-monitor/backups/`

## 15. 推荐你每次发版都按这个固定流程走

### 第一步：开发

- 在源码目录改代码

### 第二步：本机验证

- Go 编译通过
- Python 检查通过
- 前端构建通过
- 必要时 Infer 编译通过
- 联调通过

### 第三步：准备发布

- 更新版本号
- 检查 `config/*.example`
- 检查 `sql/`

### 第四步：打包

- 运行 `./scripts/package-release.sh <version>`

### 第五步：交付

- 复制到目标机

### 第六步：部署

- 运行 `./scripts/install.sh`
- 首次运行 `./scripts/init-db.sh`
- 安装并启动 systemd
- 配置 nginx
- 跑 `./scripts/health-check.sh`

### 第七步：升级

- 新版本目录执行 `./scripts/upgrade.sh`

### 第八步：回滚

- 执行 `./scripts/rollback.sh <old-version>`

## 16. 新手最容易犯的错误

### 错误 1：在 `ai-monitor-release/` 里直接改业务代码

不推荐。

应该去改真正的源码目录：

- `ai-monitor-backend/`
- `ai-monitor-service/`
- `ai-monitor-frontend/`
- `infer-server/infer-server/`

### 错误 2：在目标机上直接改 `/opt/ai-monitor/current` 里的文件

不推荐。

因为下一次升级会覆盖它。

正确做法：

- 改配置去 `/etc/ai-monitor/`
- 改程序回开发机改源码，再重新打包发版

### 错误 3：没在开发机测通就直接发版

会导致到了现场才发现：

- 前端 build 失败
- Go 编译失败
- Python 依赖不完整
- Infer 不能运行

### 错误 4：升级时把数据库覆盖掉

不应该直接手工复制旧数据库模板覆盖现场库。

正确做法：

- 数据库放在 `/var/lib/ai-monitor/`
- 升级前先备份
- 用升级脚本切版本

## 17. 你现在最实用的操作顺序

如果你是第一次正式做发布，我建议你按这个顺序练一遍：

1. 在开发机随便做一个小改动
2. 本机把功能测通
3. 更新版本号，例如 `2026.04.10-rk3576-v2`
4. 运行 `./scripts/package-release.sh 2026.04.10-rk3576-v2`
5. 检查 `dist/ai-monitor-release-2026.04.10-rk3576-v2/`
6. 把这个目录复制到另一台测试机
7. 在测试机执行 `install.sh`
8. 首次执行 `init-db.sh`
9. 配置 `/etc/ai-monitor/`
10. 安装并启动 systemd
11. 配置 Nginx
12. 执行 `health-check.sh`
13. 做一次人工验收
14. 再做一次升级和回滚演练

当你这套流程走顺以后，后续每次发版就会非常清晰。
